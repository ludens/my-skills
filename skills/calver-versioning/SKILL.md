# CalVer Versioning

## 개요

CalVer는 릴리스 날짜를 버전의 중심에 둔다. 이 스킬에서는 첫 릴리스까지 포함해 모든 릴리스에 순번을 붙이는 `YYYY.MM.DD.N` 형식을 기본으로 사용한다.

## 버전 형식

기본 형식은 다음과 같다.

```text
YYYY.MM.DD.N
```

`N`은 해당 날짜의 릴리스 순번이다. 첫 릴리스는 `0`부터 시작하고, 같은 날짜에 다시 릴리스하면 하나씩 올린다.

| 상황 | 버전 |
| --- | --- |
| 2026-05-01 첫 릴리스 | `2026.05.01.0` |
| 같은 날짜 두 번째 릴리스 | `2026.05.01.1` |
| 같은 날짜 세 번째 릴리스 | `2026.05.01.2` |

날짜는 프로젝트에 명시된 시간대를 따른다. 정해진 기준이 없으면 사용자 또는 릴리스 담당자의 현재 지역 날짜를 쓴다. 날짜가 달라질 수 있는 상황에서 UTC를 조용히 선택하지 않는다.

## 작업 절차

1. `pyproject.toml`, `package.json`, `deno.json`, `Cargo.toml`, Git 태그, README, 릴리스 노트에서 기존 버전 형식을 확인한다.
2. 프로젝트가 이미 다른 CalVer 형식을 쓰고 있다면, 사용자가 마이그레이션을 요청하지 않는 한 기존 형식을 유지한다.
3. 현재 날짜 또는 사용자가 지정한 릴리스 날짜를 기준으로 삼는다.
4. 같은 날짜의 기존 태그와 패키지 버전을 찾는다.
5. 해당 날짜의 릴리스가 없으면 `YYYY.MM.DD.0`을 쓴다. 이미 있으면 다음 `N` 값을 고른다.
6. 공개되는 모든 버전 표면을 함께 수정한다.
7. 호환성 영향은 버전 번호와 별도로 설명한다. CalVer 자체는 breaking change 여부를 표현하지 않는다.

## 확인할 파일

| 생태계 | 주로 확인할 위치 |
| --- | --- |
| Python | `pyproject.toml`, `setup.cfg`, `__version__` |
| Node | `package.json`, 패키지 매니저가 갱신하는 lock 파일 |
| Deno | `deno.json`, `jsr.json` |
| Rust | `Cargo.toml`, 필요하면 `Cargo.lock` |
| 문서 | README 설치 예시, changelog, 릴리스 노트 |
| Git | `v2026.05.01.0` 같은 태그 |

가능하면 구조화된 도구를 사용한다. 직접 수정할 때는 버전 필드와 관련 문서만 좁게 고친다.

## 같은 날짜 릴리스 규칙

`N`을 정하기 전에 기존 버전을 확인한다.

```bash
git tag --list 'v2026.05.01.*'
rg '2026\.05\.01\.'
```

일치하는 버전이 없으면 `2026.05.01.0`을 쓴다. `.0`이 있으면 `.1`, `.1`이 있으면 `.2`를 쓴다. 저장소가 이미 시간 기반 suffix를 쓰는 경우가 아니라면 시각을 버전에 넣지 않는다.

## SemVer에서 전환할 때

SemVer를 CalVer로 바꿀 때는 다음을 지킨다.

- 최종 보고에 이전 버전과 새 버전을 함께 남긴다. 예: `0.4.2 -> 2026.05.01.0`
- 변경 성격을 major/minor/patch로 억지 매핑하지 않는다.
- breaking change가 있으면 별도 호환성 메모와 마이그레이션 안내를 쓴다.
- 패키지 레지스트리 제약을 확인한다. 일부 생태계는 CalVer처럼 보이는 버전을 받아들이지만, 특이한 suffix를 정규화하거나 거부할 수 있다.

## 흔한 실수

| 실수 | 처리 |
| --- | --- |
| CalVer를 SemVer처럼 해석함 | 변경 위험과 호환성 영향을 별도로 설명한다. |
| 오늘 버전을 다시 사용함 | 같은 날짜의 다음 `.N` 값을 쓴다. |
| 날짜 형식이 섞임 | 기존 형식을 유지하거나 모든 표면을 의도적으로 함께 이전한다. |
| 한 파일만 수정함 | 이전 버전과 새 버전을 저장소 전체에서 검색한다. |
| 시간대 기준을 숨김 | 날짜가 애매하면 기준 시간대를 최종 보고에 적는다. |

## 최종 보고

마지막 응답에는 다음을 짧게 포함한다.

- 선택한 버전과 날짜 기준
- 수정한 파일
- 발견한 이전 버전
- 호환성 메모, 특히 breaking change 여부
- 실행한 검증 명령
