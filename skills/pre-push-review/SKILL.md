# Pre Push Review

푸시 직전 변경 사항을 점검해 테스트 품질과 릴리스 버전 상태를 정리한다. 사용자가 "푸시 전 확인", "배포 전 점검", "버전 올릴지 봐줘"처럼 직접 요청하면 이 절차를 따른다. 또한 사용자가 `git push`, `gh pr merge`, 배포, 릴리스, 태그 생성처럼 원격에 변경을 내보내는 작업을 하려는 경우에도 먼저 이 절차를 적용한다.

## 사용 시점

- 명시적 요청: "푸시 전 확인", "배포 전 점검", "릴리스 전에 봐줘", "버전 올릴지 봐줘"
- 암시적 상황: 사용자가 푸시, PR 머지, 릴리스, 배포, 태그 생성, publish를 진행하려고 할 때
- 직접 `git push`를 실행하기 전: 먼저 이 스킬의 점검을 끝낸 뒤 push를 진행한다
- 이미 점검한 뒤 변경이 추가로 생겼다면 push 전 다시 점검한다

## 작업 순서

1. 작업 트리와 변경 범위를 확인한다.

```bash
git status --short
git ls-files --others --exclude-standard
git diff --stat
git diff
git diff --staged
```

- `git status --short`와 `git ls-files --others --exclude-standard` 출력에서 추적되지 않은 파일을 확인한다.
- 새로 만든 소스, 테스트, README.md 같은 문서, 설정, lock 파일, 스냅샷, 생성물 중 커밋에 포함되어야 하는 파일이 빠졌는지 점검한다.
- 의도적으로 제외할 파일이면 `.gitignore`에 이미 맞게 반영되어 있는지 확인하고, 불확실하면 사용자에게 포함/제외 선택지를 묻는다.

2. 테스트 변경을 집중 검토한다.

- 새 테스트와 수정된 테스트가 실제 실패 가능한 조건을 검증하는지 확인한다.
- 항상 참인 단언, 단순 실행만 확인하는 테스트, 구현과 무관한 스냅샷 갱신, mock 호출만 반복하는 테스트를 의심한다.
- 버그 재현이나 공개 동작 보장 없이 커버리지 숫자만 올리는 테스트를 제거 대상으로 본다.
- 테스트가 의미 있는 회귀 방지라면 남기고, 왜 필요한지 짧게 설명한다.

3. 불필요한 테스트를 발견하면 제거한다.

- 제거 전후로 같은 동작 보장이 남는지 확인한다.
- 테스트 삭제가 실제 요구사항 삭제가 되지 않도록 관련 구현과 문서를 함께 본다.
- 확실하지 않으면 제거하지 말고 사용자에게 근거와 선택지를 짧게 제시한다.

4. 버전 필드를 확인한다.

- `pyproject.toml`: `[project].version`, tool별 version 필드, lock/build metadata를 확인한다.
- `package.json`: `version` 필드를 확인한다.
- 워크스페이스면 각 패키지 설정의 버전도 함께 확인한다.
- 설정 파일이 여러 개면 어떤 파일에 버전이 있는지 사용자에게 요약한다.

5. 버전업 여부를 사용자에게 확인한다.

사용자에게 아래 정보를 포함해 묻는다.

- 현재 버전
- 변경 성격 기준 추천: patch, minor, major, 또는 버전업 불필요
- 버전을 올리면 수정할 파일 목록

사용자가 버전을 올리라고 답한 경우에만 버전을 수정한다. 사용자가 명시하지 않았으면 버전을 임의로 올리지 않는다.

6. 검증을 실행한다.

- 프로젝트의 기존 테스트, 린트, 타입체크 명령을 우선 사용한다.
- 명령을 찾기 어려우면 `pyproject.toml`, `package.json`, README, CI 설정에서 확인한다.
- 실행하지 못한 검증이 있으면 이유를 최종 응답에 남긴다.

## 버전 판단 기준

- `patch`: 버그 수정, 내부 안정화, 테스트 정리, 문서 보정
- `minor`: 하위 호환 기능 추가, 공개 옵션 추가
- `major`: 깨지는 변경, 공개 API 제거, 마이그레이션 필요한 동작 변경
- `불필요`: 푸시 전 정리, CI 수정, 비릴리스 문서, 테스트만의 의미 없는 삭제

## 출력

최종 응답에는 아래만 간단히 포함한다.

- 제거한 불필요 테스트가 있으면 파일과 이유
- 추적에서 빠진 파일 점검 결과와 필요한 조치
- 버전 확인 결과와 사용자 결정
- 수정한 버전이 있으면 이전 버전과 새 버전
- 실행한 검증 명령과 결과
